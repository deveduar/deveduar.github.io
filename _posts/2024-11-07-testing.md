### 🤖 Test Automation, CI/CD y DevOps
- Ejecución automática de pruebas (unitarias, integración, E2E).  
- Se integra dentro de pipelines de [CICD](/devops/cicd/).  
- Herramientas: Jest, Cypress, [Playwright](/testing/playwright/), Selenium, [JMeter](/testing/jmeter/), etc.
- **Builds y análisis automatizados** → ejecución continua y validación del código tras cada commit.  
- **Plugins de testing** → permiten testear módulos o dependencias específicas.  
- **Scaffolding tests** → generación automática de plantillas de pruebas al crear componentes o módulos. (usado en frameworks cli)
- Integración de tests en pipelines (GitHub Actions, GitLab CI, Jenkins, CircleCI).
- Ejecución de tests en contenedores ([Docker](/software%20engineering/docker/), [Kubernetes](/virtualizacion/kubernetes/)).
- Feature Flags y Canary Releases →  activación controlada de funciones nuevas para probar en producción.
- Monitoring Feedback Loop → ciclo continuo test–build–deploy–observe.

### 💡 Metodologías, estrategias y prácticas de Testing

- [TDD - Test Driven Development](/testing/tdd---test-driven-development/) (Test Driven Development) → primero se escriben los tests.  
- [BDD](/testing/bdd/) (Behavior Driven Development) → enfoque en el comportamiento y lenguaje natural.
- [E2E - End to End Testing](/testing/e2e---end-to-end-testing/)
- **Testing Pyramid** → prioriza tests unitarios > integración > E2E.
- **Testing Trophy (Kent C. Dodds)** → equilibrio moderno de capas. (tipos de test.)
- **Regression Testing** → asegura que cambios no rompan lo anterior.
- **Exploratory Testing** → pruebas manuales no predefinidas.
- **Acceptance Testing (UAT)** → validación por parte del negocio.
- **Snapshot Testing** → compara salidas esperadas y actuales.
- **Smoke / Sanity / Snapshot tests** → capas rápidas de verificación.
- **Agile Testing** → pruebas continuas e iterativas en ciclos ágiles.  
- **CI antes del push** → ejecutar tests automáticamente en cada commit (lint, unit, integration).

### ⚙️ Test de Integración
- Validan la interacción entre módulos o servicios (backend, DB, APIs).
- Se ejecutan dentro de [CICD](/devops/cicd/).
- **Business logic y data sources** → verifican coherencia entre capas de datos y lógica empresarial.  
- **Repository pattern** → se testean las interfaces que conectan con la base de datos.  
- Ejemplo: comprobar que un microservicio devuelve correctamente datos del otro.
- Contract Testing → asegura que los microservicios cumplan sus contratos (ej: Pact).
- Service Virtualization → simula dependencias (WireMock, Hoverfly). ==tambien servicios?==
- Testcontainers → entornos efímeros (DB, Redis) para pruebas reales.
- Docker Compose → integración de entornos locales de testing.

## 🧰 QA y Monitoreo
- [QA](/testing/qa/)
- [monitoreo](/monitoreo/monitoreo/)
- **Logging y métricas** → recolectar logs y tiempos de ejecución durante pruebas.  
- **Refactoring con tests** → garantizar que los cambios no rompen el comportamiento existente.  
- **Legacy code testing** → introducir pruebas gradualmente en sistemas antiguos.  
- **Production validation** → pruebas y monitoreo en entornos reales post-deploy.

### 🧾 Gestión e informes [QA](/testing/qa/)
- **Documentación automática** → genera reportes o descripciones a partir de los tests (ej. Swagger, Allure, Jest Docs).

- Allure Report → informes visuales de test y métricas.
- TestRail / Xray → gestión de casos de prueba.
- Visual Regression Testing → detección automática de cambios visuales (Percy, Applitools).
- Accessibility Testing → pruebas de accesibilidad (axe-core, Lighthouse CI).

### 📊 Informes de calidad y cobertura
- Coveralls Docs- → mide el porcentaje de código cubierto por tests.  
  (Diferente de [Sonarqube](/devops/sonarqube/), que analiza calidad y seguridad del código.)
- SonarQube → análisis estático (bugs, seguridad, mantenibilidad).
- Codecov → cobertura visual, alternativa moderna a Coveralls.
- Mutation Testing → evalúa la robustez de los tests (StrykerJS, PIT).
- ESLint / Prettier → linting y estilo como parte del pipeline de calidad.

### ⚡ Test de rendimiento, carga y seguridad
- [JMeter](/testing/jmeter/) → medir rendimiento, carga concurrente y estrés del sistema.  
- Evalúa escalabilidad y tiempo de respuesta bajo carga.

- k6 → alternativa moderna a [JMeter](/testing/jmeter/) para rendimiento (JS-based).
- Lighthouse → performance, accesibilidad y métricas web.
- OWASP ZAP / [Burp Suite](/ciberseguridad/burp-suite/) → pruebas de seguridad automatizadas.
- Chaos Testing / Gremlin → resiliencia ante fallos.
- Smoke Testing / Sanity Testing → validaciones rápidas post-deploy.
- **Time-based tests** → validan comportamientos dependientes del tiempo (ej. expiración de tokens, tareas programadas).  
- **Benchmark testing** → mide duración o rendimiento de ejecuciones unitarias.
- **Time performance testing** → mide tiempos de ejecución y latencias internas.

## 🔄 Patrones de Comportamiento y Dependencias

- Dobles en Test - Diferencia entre Stub, Fake y Mock-stub-fake-mock  
- ejemplos de mocks stub y fakes
### 🧱 Fakes
- funcional  
- dependencias (ej. DB)  
- sistemas complejos  
- verificar comportamiento  
- fakes para reactividad mejor  
- ej: sistema de inventario  

### 🧱 Mocks
- parámetros, métodos, número de llamadas  
- valida llamadas  
- dependencias  
- ej: solicitud de correo  
- [MSW Mocks service worker](/testing/msw-mocks-service-worker/)  

### 🧱 Stubs
- dependencias, control de estado  
- sin lógica, devuelven valores predefinidos  
- comportamientos o datos predefinidos  
- ej: devuelve usuario
