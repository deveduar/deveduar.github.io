Los **cron jobs** son una pieza fundamental en sistemas Unix/Linux para la **ejecuciГіn automГЎtica de tareas programadas**. Se integran de forma natural en flujos de **automatizaciГіn**, **operaciones backend** y **DevOps**, permitiendo ejecutar procesos sin intervenciГіn humana en momentos especГӯficos o de forma recurrente.

## ВҝQuГ© es cron?
**cron** es un **demonio del sistema** que se ejecuta en segundo plano y se encarga de lanzar tareas (jobs) segГәn una planificaciГіn definida por el usuario.

- Funciona a nivel de sistema operativo.
- Lee configuraciones desde archivos llamados **crontabs**.
- Es extremadamente ligero, estable y disponible por defecto en la mayorГӯa de distribuciones Linux y Unix.

## ВҝQuГ© es un cron job?
Un **cron job** es una **tarea programada** que cron ejecuta automГЎticamente cuando se cumple una condiciГіn temporal.

- Puede ejecutar:
	- Comandos de shell
	- Scripts (bash, python, node, php, etc.)
	- Binarios del sistema
- Se define mediante una **expresiГіn de tiempo + comando**
- Puede pertenecer a:
	- Un usuario concreto
	- El sistema (cron del sistema)

## Casos de uso habituales
- Backups automГЎticos de bases de datos
- Limpieza de archivos temporales o logs
- EjecuciГіn de ETLs y procesos batch
- EnvГӯo de emails programados
- SincronizaciГіn de datos con APIs externas
- Tareas de mantenimiento (rotaciГіn de logs, chequeos)
- Builds nocturnos o tareas de integraciГіn
- EjecuciГіn de scripts de scraping periГіdicos

## Crontab

## Variables de entorno en cron
Los cron jobs **no cargan el entorno completo del usuario**, lo que implica:

- PATH limitado
- Variables de entorno no disponibles
- No se ejecutan `.bashrc` o `.profile`

Buenas prГЎcticas:
- Usar rutas absolutas (`/usr/bin/python`)
- Definir variables explГӯcitamente
- Cargar un script que inicialice el entorno

Ejemplo:
{% raw %}
```

```

## Permisos y seguridad
- El cron job se ejecuta con los **permisos del usuario** propietario
- Es crГӯtico:
	- Validar scripts
	- Limitar permisos de escritura
	- Evitar ejecuciГіn como root salvo necesidad

- `/etc/cron.deny`

## Cron del sistema vs cron de usuario
### Cron de usuario
- Configurado con `crontab -e`
- MГЎs sencillo
- Ideal para tareas personales o de aplicaciones

### Cron del sistema
- Definido en `/etc/crontab` o `/etc/cron.d`
- Incluye campo adicional de usuario
- Usado para tareas administrativas

## Cron y DevOps
En entornos modernos:
- Se usa cron para:
	- Jobs de mantenimiento
	- OrquestaciГіn simple
- Se complementa o sustituye por:
	- Kubernetes CronJobs
	- CI/CD schedulers
	- Sistemas de colas y workers

Cron sigue siendo:
- Fiable
- Predecible
- FГЎcil de auditar

## Limitaciones de cron
- No gestiona dependencias entre jobs
- No tiene control de estado ni reintentos
- No es adecuado para flujos complejos
- DifГӯcil de escalar horizontalmente

## Buenas prГЎcticas
- Documentar cada cron job
- Usar scripts versionados
- AГұadir logs y alertas
- Evitar solapamiento de ejecuciones
- Probar manualmente antes de automatizar
- Revisar periГіdicamente jobs obsoletos

## Estado de cron en 2025
En **2025**, cron sigue siendo una **herramienta vigente, estable y ampliamente usada** en sistemas Unix/Linux. A pesar del auge de plataformas cloud, orquestadores y sistemas distribuidos, cron mantiene su relevancia por su **simplicidad, fiabilidad y bajo coste operativo**.

- Presente por defecto en prГЎcticamente todas las distribuciones Linux
- MГӯnima sobrecarga de recursos
- Comportamiento predecible y bien conocido
- Ideal para:
	- Servidores tradicionales
	- VPS
	- MГЎquinas on-premise
	- Automatizaciones simples y medianas

Cron no ha cambiado conceptualmente, pero **su rol ha evolucionado** dentro de arquitecturas modernas.

## Cron en arquitecturas modernas
En 2025 cron suele ocupar uno de estos roles:

- AutomatizaciГіn local en servidores
- OrquestaciГіn bГЎsica de scripts
- Lanzador de jobs que delegan trabajo a:
	- APIs
	- Colas de mensajes
	- Workers distribuidos
- Soporte a pipelines legacy

En muchos equipos:
- **cron = capa base**
- Herramientas modernas = capas superiores

## Ventajas que lo mantienen vigente
- Curva de aprendizaje mГӯnima
- No depende de terceros
- FГЎcil de versionar indirectamente (infra como cГіdigo)
- Ideal para tareas deterministas
- Muy difГӯcil de romper

## Limitaciones frente a soluciones modernas
- Sin gestiГіn nativa de:
	- Retries
	- Estados
	- Dependencias
	- Observabilidad avanzada
- Escalado limitado
- DifГӯcil de usar en entornos efГӯmeros (containers)

Estas limitaciones explican la apariciГіn de **alternativas especializadas**.
